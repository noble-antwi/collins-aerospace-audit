# Contributing to Collins Aerospace IT Audit Project

Thank you for your interest in this academic project. This repository serves as an educational resource for IT auditing, cybersecurity, and risk management studies.

## Purpose

This repository is maintained as an academic resource demonstrating:
- Comprehensive IT audit documentation standards
- Application of multiple security frameworks (COBIT, NIST, CMMC)
- Post-incident analysis and remediation planning
- Professional documentation practices
- Executive communication strategies

## Types of Contributions

### Welcome Contributions

We welcome contributions that:

1. **Correct Technical Errors**
   - Framework references or standard citations
   - Control mapping inaccuracies
   - Technical terminology corrections

2. **Improve Documentation**
   - Clarify complex concepts
   - Add cross-references
   - Improve markdown formatting
   - Fix typos and grammatical errors

3. **Add Educational Value**
   - Additional framework mappings
   - Supplementary analysis
   - Alternative risk assessment approaches
   - Expanded remediation strategies

4. **Enhance Structure**
   - Improve navigation
   - Add useful indexes or glossaries
   - Better organization of materials

### Out of Scope

Please do not submit contributions that:
- Alter the core findings or conclusions (this is a completed academic project)
- Add actual company confidential information
- Include unsupported claims about Collins Aerospace
- Introduce non-academic content

## Contribution Process

### For Students and Educators

If you're using this as a reference for your own work:

1. **Fork the Repository**
   - Create your own fork to use as a template
   - Adapt the structure for your own case studies
   - Give appropriate credit if you use significant portions

2. **Cite Properly**
   - Include proper academic citations
   - Reference specific sections you've adapted
   - Follow your institution's academic integrity policies

### For Technical Corrections

1. **Open an Issue First**
   - Describe the error or improvement
   - Provide references supporting your correction
   - Explain the impact of the issue

2. **Submit a Pull Request**
   - Reference the issue number
   - Keep changes focused and specific
   - Include supporting documentation
   - Update any affected cross-references

### For Framework Updates

As security frameworks evolve:

1. **Document Version Changes**
   - Note when frameworks are updated (e.g., NIST SP 800-53 Rev. 6)
   - Add changelog entries
   - Maintain backward compatibility notes

2. **Preserve Historical Context**
   - This audit was conducted in Fall 2025
   - Framework versions current at that time should be primary
   - Add supplementary notes about newer versions

## Documentation Standards

### Writing Style

- **Professional tone** - No emojis, informal language, or slang
- **Clear and concise** - Academic but accessible
- **Well-structured** - Use headings, lists, and tables effectively
- **Properly cited** - APA 7th Edition format for all references

### Markdown Formatting

- Use consistent heading hierarchy (# for main titles, ## for sections, etc.)
- Include blank lines between sections
- Use tables for structured data
- Include code blocks with appropriate language tags
- Ensure all links are functional

### File Naming

- Use lowercase with hyphens: `file-name.md`
- Be descriptive but concise
- Include version numbers where appropriate
- Follow existing naming conventions

## Review Process

### Student/Educational Contributions

All educational contributions will be reviewed by the project maintainers:
- Noble Antwi (original project lead)
- Academic advisors at Illinois Institute of Technology

Review criteria:
- Technical accuracy
- Educational value
- Consistency with existing documentation
- Proper citation and attribution

### Timeline

- Initial review: Within 2 weeks
- Feedback provided: Within 3 weeks
- Final decision: Within 4 weeks

Complex contributions may require additional review time.

### Expected Behavior

## Version Control

### Branching Strategy

- `main` - Stable, reviewed content
- `develop` - Accepted changes pending final review
- `feature/*` - Specific contribution branches

### Commit Messages

Use clear, descriptive commit messages:
- `docs: Correct NIST 800-53 control reference in Finding #1`
- `structure: Reorganize remediation timeline section`
- `fix: Update broken links in References section`

### Version Numbers

This project follows semantic versioning:
- MAJOR.MINOR.PATCH
- Major: Significant structural changes
- Minor: Content additions or improvements
- Patch: Corrections and minor fixes
